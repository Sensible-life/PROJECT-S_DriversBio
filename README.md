# DriveAI

운전 중에도 손을 쓰지 않고 공부할 수 있도록 만드는 음성 기반 학습 시스템.

이 프로젝트의 핵심은 `주입식 오디오 재생`이 아니라, 실제로 `말하면서 암기하고`, `설명하고`, `검증받는` 학습 루프를 차 안에서 돌리는 것이다.

## 1. 프로젝트 목표

### 문제 정의
- 운전 중에는 화면을 보기 어렵고 손으로 조작하기도 어렵다.
- 일반 오디오 강의는 수동 청취에 그치기 쉬워 암기 효율이 낮다.
- 책, 노트, 강의 요약본 같은 텍스트 학습 자료를 능동적으로 복습할 수 있는 구조가 필요하다.

### 목표
- 학습 자료를 넣어두면 음성 기반으로 퀴즈를 내고 답하게 한다.
- 사용자가 말로 설명하면 부족한 부분을 음성으로 보완해준다.
- 사용자의 설명 내용이 사실과 맞는지 검증해준다.
- 틀린 개념, 약한 개념을 누적 관리해서 다음 세션에 다시 출제한다.

## 2. 핵심 사용자 경험

사용자는 차에 탑승한 뒤 이어폰이나 차량 블루투스로 시스템을 실행한다.

시스템은 다음처럼 동작한다.

1. 오늘 복습할 자료를 선택하거나 자동 추천한다.
2. 음성으로 문제를 낸다.
3. 사용자가 말로 답한다.
4. 시스템이 답변을 분석한다.
5. 맞은 부분, 틀린 부분, 빠진 부분을 음성으로 피드백한다.
6. 필요하면 추가 질문을 던진다.
7. 세션 종료 후 약점 개념을 저장하고 다음 학습에 반영한다.

## 3. 핵심 기능

### A. 학습 자료 입력
- 텍스트 직접 입력
- Markdown/메모 업로드
- 책/강의/렉처 노트 요약본 붙여넣기
- 웹 콘텐츠 저장

### B. 학습 모드
- 퀴즈 모드: 개념 질문, 빈칸 질문, 비교 질문, 사례 질문
- 설명 모드: 사용자가 주제를 설명하면 시스템이 누락 내용을 채움
- 검증 모드: 사용자의 설명이 원문/정답 기준과 얼마나 일치하는지 판정
- 반복 암기 모드: 짧은 개념을 빠르게 순환 복습

### C. 음성 인터랙션
- 음성 명령으로 시작/중지/다음 문제/다시 설명
- STT 기반 답변 인식
- TTS 기반 피드백 재생
- 끊김 없이 이어지는 짧은 대화형 세션

### D. 학습 관리
- 약한 개념 자동 태깅
- 틀린 이유 저장
- 복습 우선순위 계산
- 세션 히스토리 저장
- 진도/정확도/설명력 추적

## 4. 제품 방향

### 우선순위
`Flutter 모바일 앱 중심`으로 간다.

이유:
- 차량 블루투스, 마이크, 백그라운드 오디오 제어는 앱이 더 안정적이다.
- 운전 중 핵심 경험은 모바일에서 완결되어야 한다.
- 웹/확장 실험 코드는 정리하고 앱 + 백엔드 구조에 집중하는 편이 구현과 유지보수 모두 단순하다.

### 권장 구성
- 모바일 앱: Flutter 기반 메인 제품. 운전 중 세션 실행, 음성 입출력, 히스토리 확인
- 백엔드: 자료 저장, 세션 관리, AI 프롬프트 오케스트레이션

## 5. 시스템 구조

## 전체 아키텍처

```text
[Mobile App]
  -> 음성 세션 실행
  -> 학습 자료 입력/조회
  -> STT/TTS 처리
  -> 차량 블루투스 오디오 연결
  -> Backend/API

[Backend/API]
  -> 사용자/콘텐츠/세션 관리
  -> AI 호출 오케스트레이션
  -> 퀴즈 생성
  -> 답변 평가
  -> 약점 분석

[AI Provider Layer]
  -> OpenAI 또는 Gemini
  -> 모델 교체 가능하도록 Provider 추상화

[Storage]
  -> 학습 자료
  -> 문제 세트
  -> 세션 로그
  -> 약점 개념
```

## 6. 주요 워크플로우

### 1) 자료 등록 워크플로우
1. 사용자가 책 요약, 강의 노트, 메모를 입력한다.
2. 시스템이 단락/개념 단위로 분해한다.
3. 핵심 개념, 정의, 비교 포인트, 암기 포인트를 추출한다.
4. 퀴즈 후보와 설명 기준답안을 생성한다.
5. 학습 세트로 저장한다.

### 2) 운전 중 학습 워크플로우
1. 사용자가 음성 명령 또는 앱 실행으로 세션을 시작한다.
2. 시스템이 오늘의 복습 후보를 선택한다.
3. TTS로 문제를 읽어준다.
4. 사용자가 음성으로 답한다.
5. STT 결과를 정규화한다.
6. LLM이 정답성, 누락 개념, 허위/오개념 여부를 판정한다.
7. 시스템이 짧고 안전한 음성 피드백을 준다.
8. 필요 시 꼬리 질문을 한다.
9. 세션 종료 후 결과를 저장한다.

### 3) 설명 검증 워크플로우
1. 사용자가 "이 개념 설명해볼게"라고 말한다.
2. 시스템이 일정 시간 동안 설명을 듣는다.
3. 설명 내용을 기준 자료와 비교한다.
4. 누락된 핵심, 잘못된 진술, 애매한 부분을 분리한다.
5. 시스템이 핵심만 음성으로 교정한다.

## 7. AI 역할 정의

### 공통 역할
- 학습 자료 구조화
- 문제 생성
- 답변 채점
- 설명 보강
- 진위 검증
- 다음 질문 선택

### OpenAI / Gemini 공통 설계 원칙
- 특정 벤더에 고정되지 않도록 Provider 인터페이스를 둔다.
- `quiz generation`, `answer evaluation`, `gap fill`, `truth check`를 각각 분리된 서비스로 구성한다.
- 세션 중에는 짧고 안정적인 응답이 중요하므로 응답 길이 제한과 안전한 출력 포맷을 강제한다.

- 정답 기준 설명
- 오답 예시
- 마지막 복습 시점
- 정답률
- 설명 완성도

## 9. MVP 범위

### 1차 MVP
- 텍스트/Markdown 학습 자료 등록
- 자료를 개념 단위로 분리
- 퀴즈 생성
- 음성 질문 재생
- 음성 답변 수집
- 답변 채점 및 짧은 피드백
- 세션 결과 저장

### 2차 확장
- 설명 모드
- 진위 검증 강화
- 약점 기반 재출제
- 운전 시작 시 자동 추천
- 자료 수집 보조 도구가 필요하면 앱 내 import 흐름 또는 별도 관리도구 추가

### 3차 확장
- 실시간 음성 채팅형 튜터
- 사용자 맞춤 학습 페르소나
- 시험 모드 / 압박 면접 모드
- 장기 기억 곡선 기반 스케줄링

## 10. 기술 제안

### 클라이언트
- 모바일 앱: Flutter

### 백엔드
- API 서버: FastAPI 또는 NestJS
- DB: PostgreSQL
- 캐시/큐: Redis
- 파일 저장: Supabase Storage 또는 S3 호환 스토리지

### AI/음성
- LLM: OpenAI / Gemini 중 선택 가능
- STT: Whisper 계열 또는 플랫폼 STT
- TTS: OpenAI TTS 또는 플랫폼 TTS

## 11. 안전 설계 원칙

운전 중 사용이므로 아래를 기본 원칙으로 둔다.

- 긴 화면 조작을 요구하지 않는다.
- 모든 핵심 기능은 음성 명령으로 수행 가능해야 한다.
- 피드백은 짧고 명확해야 한다.
- 장문 설명은 세션 종료 후 요약으로 넘긴다.
- 주행 중에는 시각 정보보다 오디오 중심으로 설계한다.
- 위험 상황에서는 자동으로 상호작용 강도를 낮출 수 있어야 한다.

## 15. 현재 구현 상태

현재 저장소에는 아래 프로토타입이 구현되어 있다.

- `services/api`: 무의존 Node HTTP API 서버
- `packages/session-engine`: 자료 분해, 퀴즈 생성, 답변 평가, 약점 집계 로직
- `packages/ai-provider`: OpenAI/Gemini 교체를 위한 Provider 레이어와 실행 가능한 mock provider
- `apps/mobile/src/demo-session.js`: 터미널 기반 모바일 세션 시뮬레이터
- `apps/web`: 브라우저에서 자료 등록과 세션 테스트를 할 수 있는 정적 프로토타입
- `apps/extension`: 선택 텍스트를 API로 보내는 크롬 확장 초안
